# “不一样的贪吃蛇”未产出代码事件 RCA

## 1. 事件摘要

- 事件时间：2026-03-11
- 触发请求：`开发一个不一样的贪吃蛇游戏，仅前端即可，不需要后端`
- 现象：系统持续输出访谈、创意、技术方案和测试规划，但没有进入实际编码，也没有生成任何代码文件。
- 最终状态：在“技术方案与架构设计”阶段发生 `500 InternalServiceError`，流程中断。

## 2. 结论摘要

这次“没有代码产出”不是由单点问题造成，而是由以下三类问题叠加导致：

1. 编排层将该请求优先路由成“多阶段协作文档流程”，没有尽早进入“直接实现 MVP”的执行路径。
2. 运行环境被绑定到空工作目录 `/Users/like/code/apptest`。空目录本身不是错误，理论上完全可以直接在该目录初始化并落代码；真正的问题是系统把它当成“缺少上下文，需要先做需求和方案”的信号，导致继续空转。
3. 在真正有机会转入开发前，架构阶段模型调用返回 `500 InternalServiceError`，而系统没有基于检查点自动续跑，导致流程直接终止。

## 3. 关键事实

### 3.1 该请求被重复编排为多个 workflow

同一用户请求被多次生成工作流，至少出现了多个 `workflowId`。这导致上下文出现重复的 owner plan、重复的 human directive 和记忆注入，系统更容易继续做“协作推进”和“文档归纳”，而不是收敛为一次明确的实现任务。

影响：

- 上下文噪声增加
- 阶段切换变得不稳定
- 更容易重复做需求分析、方案设计、测试规划

### 3.2 空工作目录不是根因，但被错误解释

日志显示工作目录多次为 `/Users/like/code/apptest`，且目录为空、`README.md` 不存在。

这本身并不阻止编码：

- 对“从零开发一个前端小游戏”来说，空目录是可接受输入
- 正确行为应当是在空目录中直接初始化最小前端项目并开始写代码

真正的问题在于：

- 代理把“空目录”解释成“需要先继续做需求和架构”
- 没有启用“greenfield 模式”或“空目录快速初始化策略”
- 因而没有触发任何 `Write/Edit` 类型的代码生成动作

### 3.3 阶段执行明显偏向访谈和文档

架构师在用户已经明确限定“仅前端、不需要后端”后，仍然输出了较长的需求访谈问卷和创意方向梳理；全栈开发专家则输出了技术可行性评估文档；测试工程师继续输出测试场景规划。

这说明调度策略更像：

- 先需求访谈
- 再方案设计
- 再测试规划

而不是：

- 根据已足够明确的需求，先实现一个可玩的 MVP
- 需要时再补充方案和测试

### 3.4 没有任何代码写入证据

日志里能看到的主要工具调用为：

- `Glob`
- `Grep`
- `Read`
- `TodoWrite`

未看到进入实现阶段后的有效 `Write/Edit` 写文件动作。说明问题不是“代码写了但丢了”，而是“流程根本没有进入编码”。

### 3.5 直接终止点是架构阶段 500

在“技术方案与架构设计”阶段，架构师任务返回 `500 InternalServiceError`。系统没有：

- 自动重试同一任务
- 将任务降级给全栈直接实现
- 从最近检查点继续

因此流程在“进入开发之前”就被中断。

## 4. 时间线

### T0 用户请求进入系统

用户明确提出：

- 开发一个不一样的贪吃蛇游戏
- 仅前端即可
- 不需要后端

按任务性质，这已经足够支持“直接在空目录初始化一个前端游戏项目并实现 MVP”。

### T1 编排层生成多阶段协作 workflow

系统将任务拆成：

- 需求澄清/创意定义
- 技术方案与架构设计
- 核心功能开发
- 测试与交付

这是合理但偏重的流程。对一个纯前端小游戏而言，这种拆法成本过高，且容易造成“文档先行、编码滞后”。

### T2 需求阶段持续扩张

架构师执行了需求访谈、创意发散；全栈补充了可行性分析；测试工程师提前输出测试规划。

此时系统已经拥有足够多的信息来开始落地一个 MVP，但没有发生阶段压缩，也没有触发“先写再迭代”。

### T3 工作目录被识别为空

多个代理发现：

- 工作目录为空
- 没有 `README.md`
- 没有现有项目结构

正确处理应是“初始化项目”；实际处理则是“继续输出分析文档”。

### T4 架构阶段发生 500

进入“技术方案与架构设计”后，架构师任务返回 `500 InternalServiceError`。

### T5 流程中止

系统未从最近可恢复状态继续，也未切换执行者，因此最终没有代码输出。

## 5. 根因拆解

### 5.1 直接原因

架构阶段发生 `500 InternalServiceError`，流程没有自动恢复。

### 5.2 近端原因

- 没有检查点恢复机制
- 没有失败后自动降级策略
- 没有在空目录场景下优先触发 greenfield 初始化

### 5.3 系统性原因

#### A. 编排策略过重

系统默认采用“需求 -> 架构 -> 开发 -> 测试”的完整协作模板，即使面对的是一个可以快速落地的小游戏需求。

结果：

- 分析链条过长
- 阶段之间依赖过强
- 任意中间阶段失败都会阻断最终产出

#### B. 缺少 greenfield 启发式

对于“空目录 + 明确需求 + 前端即可”的组合，系统没有识别为高置信度 greenfield 场景。

建议的期望行为应为：

- 直接初始化 Vite/React 或原生 Canvas 工程
- 先交付可玩原型
- 再补玩法创新、样式和测试

#### C. 缺少“足够明确即可编码”的收敛规则

用户需求虽然抽象，但已经满足“先做 MVP，再逐步细化”的条件。系统仍然把“不一样”当成必须访谈清楚的前置条件。

这会导致：

- 将开放性需求错误地理解为阻塞条件
- 让架构产出替代了编码产出

#### D. 缺少任务级断点续跑

当模型调用失败时，系统没有记录“已完成到哪一步、下一步该做什么、谁可以接替”，导致一次 500 直接终止整条链路。

## 6. 为什么“空目录”不是问题本身

需要明确校正一个判断：

- “空目录”不是事故原因
- “空目录但未触发初始化”才是问题

对于这类请求，空目录恰恰意味着：

- 不存在兼容性包袱
- 不需要读懂旧代码
- 更适合一键初始化和直接实现

因此后续规则不应是“空目录 -> 继续分析”，而应是：

- 空目录 + 新项目请求 -> 初始化项目并开始编码

## 7. 断点续跑方案

下面给出一个可落地的“检查点 + 自动恢复”方案，用于处理类似 500 崩溃。

### 7.1 断点粒度

按 `workflow -> stage -> task -> action` 四级持久化检查点。

每次满足以下条件之一时写入检查点：

- 一个 task 开始前
- 一个 task 完成后
- 一次关键工具调用前后
- 阶段切换前后

### 7.2 检查点最小字段

建议持久化以下内容：

```json
{
  "workflowId": "81084769-4fd8-4ac4-87f2-8e4acd5ada0d",
  "stageId": "49fe8a37-951c-4028-bf20-4bb96cef3073",
  "stageTitle": "技术方案与架构设计",
  "taskId": "b540591a-b4a7-4636-9fec-ecf19c98540a",
  "taskTitle": "设计前端技术架构与模块划分",
  "assignee": "架构师",
  "workingDirectory": "/Users/like/code/apptest",
  "repoSnapshot": {
    "entries": [],
    "isEmpty": true
  },
  "artifacts": [
    "创意列表",
    "技术可行性评估",
    "测试场景规划"
  ],
  "todos": [
    {"id": "1", "content": "确定技术选型", "status": "in_progress"},
    {"id": "2", "content": "设计核心模块架构", "status": "pending"},
    {"id": "3", "content": "定义模块接口", "status": "pending"},
    {"id": "4", "content": "输出架构文档", "status": "pending"}
  ],
  "resumeHint": "空目录，可直接切换为 greenfield 初始化并实现 MVP",
  "updatedAt": "2026-03-11T21:xx:xx+08:00"
}
```

### 7.3 恢复策略

#### 策略 A：同任务重试

适用条件：

- 首次 500
- 上下文长度未超阈值
- 当前 assignee 可继续执行

执行方式：

1. 读取最近检查点
2. 保留 `workflowId/stageId/taskId`
3. 将最近一次成功输出摘要化
4. 重试当前 task，不重复注入完整长历史

#### 策略 B：同阶段降级切换执行者

适用条件：

- 连续 2 次 500
- 当前 task 偏方案型，但已有足够信息

执行方式：

1. 读取最近检查点
2. 把 `resumeHint` 改写为“跳过剩余架构细化，直接实现 MVP”
3. 将 task 转交给全栈开发专家
4. 明确限制：只做最小实现，不等待更多访谈

这次事件最适合走策略 B。

#### 策略 C：阶段跳过

适用条件：

- 当前阶段不是硬前置
- 用户需求已足够明确
- 项目为 greenfield

执行方式：

1. 标记当前 stage 为 `degraded_completed`
2. 将已有方案摘要作为开发输入
3. 直接进入“核心功能开发”

### 7.4 针对本事件的推荐续跑断点

建议把断点设在：

- `workflowId = 81084769-4fd8-4ac4-87f2-8e4acd5ada0d`
- `stageTitle = 技术方案与架构设计`
- `taskTitle = 设计前端技术架构与模块划分`

但恢复时不要继续要求架构师补全文档，而应直接执行：

1. 读取已产出的创意共识：`道具系统 + 赛博朋克视觉风格 + 基础关卡模式`
2. 识别工作目录为空，触发 greenfield 初始化
3. 由全栈开发专家直接在 `/Users/like/code/apptest` 创建最小前端项目
4. 先落地一个可玩 MVP：
   - Canvas 渲染
   - 基础蛇移动
   - 食物与得分
   - 2 到 4 种道具
   - 赛博风格 HUD
5. 成功生成文件后，再补回轻量设计文档和测试

## 8. 修复建议

### 8.1 编排层修复

- 对“空目录 + 新项目 + 纯前端/纯后端 + 无现有代码依赖”的请求启用 greenfield 快速路径
- 将默认阶段数压缩为“实现优先”
- 若用户未明确要求详细方案，不要强制先走完整需求访谈

### 8.2 恢复机制修复

- 为每个 stage/task 写入检查点
- 对 500 启用自动重试
- 连续失败后自动降级执行者或跳阶段

### 8.3 上下文治理修复

- 同一用户请求只允许一个活跃 workflow
- 检测到重复 workflow 时进行合并
- 减少重复注入相同 `Human directive`

### 8.4 开发触发规则修复

- 若日志中连续出现文档型交付，且尚无任何 `Write/Edit`，应触发告警
- 达到阈值后自动转入“开始写最小可运行代码”

## 9. 可执行 SOP

针对未来同类请求，建议系统按以下 SOP 处理：

1. 判断是否为 greenfield 项目。
2. 若工作目录为空且需求足够明确，直接初始化项目。
3. 在第一次结构化分析后，必须产出至少一个代码文件。
4. 若连续两个阶段都只有文档产出且无代码写入，强制切换到实现模式。
5. 若发生 500：
   - 优先从最近检查点重试
   - 再失败则切换 assignee
   - 仍失败则跳过非关键阶段，直接进入实现

## 10. 最终结论

这次事故的本质不是“模型不会写贪吃蛇”，也不是“空目录导致无法编码”，而是：

- 编排系统把一个适合快速实现的任务过度流程化；
- 空目录场景没有触发正确的 greenfield 初始化策略；
- 中间阶段出现 500 后没有检查点恢复和降级续跑；
- 最终导致流程在进入实际编码前就终止。

如果只修 500 重试，而不修编排和 greenfield 策略，后续仍然可能继续“写很多文档但不写代码”。如果只修编排，不修检查点，流程仍然会被单次 500 打断。三者需要一起处理。
