# 群聊记忆最佳实践

## 目标
- 让群聊在多轮协作中持续保留关键上下文，而不是每次从零开始。
- 控制记忆质量和体量，避免“垃圾记忆”污染后续任务。

## 推荐分层
- `Pinned Charter`（长期、置顶）:
  - 每个群聊创建时写入 1 条固定记忆，包含群目标、群主职责、协作机制。
  - `scope=work_group`，`pinned=true`，`ttl=null`。
- `Directive Memory`（中期、可过期）:
  - 每次人类消息写入 1 条“指令摘要记忆”。
  - `scope=work_group`，`tags=["human","directive","chat_memory"]`，建议 `ttl=7~30 天`。
- `Execution Summary`（中长期、可检索）:
  - 每次任务执行结束后写入执行摘要到 `work_group`（可选同时写入 `agent/user`）。
  - 用于后续任务注入上下文与复盘。

## 质量门槛
- 只写“可复用信息”:
  - 约束、决策、里程碑、失败原因、下一步行动。
- 避免写“瞬时噪声”:
  - 聊天寒暄、无结论争论、低价值中间过程。
- 单条长度控制:
  - 建议不超过 200~300 字；超长内容先摘要再入库。

## 注入策略
- 优先级:
  1. pinned 记忆
  2. 最近的 directive 记忆
  3. 最近的执行摘要
- 容量:
  - 任务上下文注入建议上限 6~10 条，优先保证“新 + 关键”。
- 作用域:
  - 默认读 `user + work_group + agent`，写至少包含 `work_group`。

## 生命周期治理
- 过期清理:
  - 对 directive 记忆启用 TTL 并定期清理。
- 去重与合并:
  - 连续重复约束可合并为一条高质量记忆，减少冗余。
- 可观察性:
  - 记录 memory 审计事件（来源、scope、数量），便于排查“为什么模型这样回答”。
